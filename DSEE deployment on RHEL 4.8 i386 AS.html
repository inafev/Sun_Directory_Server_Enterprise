Platform Support

In Directory Server Enterprise Edition, support for Windows 2000, Red Hat Advanced Server 3.0, and J2SE platform 1.4 has been removed. Support for the native install package releases for platforms other than the Solaris operating system has been removed. Support for 32–bit versions of the software might be discontinued for some platforms. To be prepared, plan the transition to 64–bit versions of the software and to newer versions of the supported operating systems. See Operating System Requirements for details of the newer versions of supported operating systems.

JDK versions 1.5 and 1.6 are supported in this release.

Identity Synchronization for Windows 6.0 Service Pack 1 can synchronize with Windows Active Directory 2008.

Directory Server Enterprise Edition Bundle Patch 6.3.1.1.2 supports Logical Domains, (LDoms), on the SPARC platform for Solaris 10 Update 3 and later versions. For more information about LDoms, see the Logical Domains (LDoms) 1.0.1 Administration Guide.

Understanding the Moved It or Copied It Option in VMWare Workstation

Whenever you start a virtual machine in VMWare Workstation that has been moved or copied you are asked to specify which one of those two things happened. This tip explains the importance of your selection.

If you’ve ever moved or copied a virtual machine and tried to launch it using VMWare Workstation then you’ve seen the dialogue that asks you whether or not the VM was moved or copied. The reason this option is presented to you is based around the concept of the virtual machines Universally Unique Identifier, or UUID. The UUID is stored in the SMBIOS for each VM. This is a 16 byte value that is based upon an identifier of the virtual host and the path of the VM, and is used to uniquely identify each VM on the virtual host.

That being the case, if you moved the virtual machine you can keep the UUID by telling VMWare that you moved it. If it was copied you should be sure and generate a new UUID by telling VMWare that the VM was copied. If two VMs are have the same UUID there could be negative results, including one of the VMs not being allowed to boot.

Configuring Network Parameters
Configuring Window Size (consumer/replica host required)

The window size (default value10) represents the maximum number of update messages that can be sent without immediate acknowledgment from the consumer.
It is more efficient to send many messages in quick succession instead of waiting for an acknowledgment after each message. Using the appropriate window size, you can eliminate the time replicas spend waiting for replication updates or acknowledgments to arrive.
If your consumer replica is lagging behind the supplier, increase the window size to a higher value than the default, such as 100, and check replication performance again before making further adjustments. When the replication update rate is high and the time between updates is therefore small, even replicas connected by a local area network (LAN) can benefit from a higher window size.
